개인 프로젝트로 투자 도우미 서비스를 만들고 있습니다. 포트폴리오를 등록하면 시세·배당 정보와 함께 AI 투자 인사이트를 제공하는 서비스입니다. 이 글은 그중 AI 기능을 서비스에 넣으며 내린 설계 결정에 대한 기록입니다.

### 어떤 서비스였나?

사용자의 투자 성향 설문과 직접 등록한 보유 종목을 바탕으로, 로컬 LLM이 투자 인사이트를 생성해 주는 기능입니다. 추론 한 번에 평균 40~50초가 걸리는 무거운 작업인데, 처음에는 코어 서비스가 LLM을 직접 호출하는 구조였습니다.

기능은 잘 돌았지만, LLM과 한집에 사는 비용이 하나둘 보이기 시작했습니다.

### ai-app을 별도 서버로 분리한 이유

1. **변경 격리** — 로컬 LLM은 모델 교체, 프롬프트 수정, 클라이언트 코드 변경이 잦습니다. 실제로 모델이 한자를 섞어 출력하는 문제로 "한국어 강제" 시스템 프롬프트를 덧대는 식의 수정이 반복됐습니다. 이런 변경을 ai-app에 가둬서, AI 쪽을 수정·재배포해도 코어 서비스는 재기동 없이 운영됩니다.

2. **장애 격리** — 별도 프로세스(JVM)라 AI 쪽 크래시·부팅 실패가 코어로 전파되지 않습니다. 분리 전에는 Spring AI 자동설정 때문에 쓰지도 않는 더미 api-key를 넣지 않으면 본체가 부팅조차 안 되는 일도 있었습니다. 분리 후에는 LLM이 죽어도 코어는 무영향이고, 인사이트는 규칙 기반 폴백으로 대응합니다.

3. **시크릿·접근 경계** — GPU 서버 주소, API 키 같은 AI 관련 비밀값이 코어 환경변수에서 제거됐고, ai-app은 외부에 노출되지 않으며 내부 인증(X-System-Key)으로만 접근됩니다.

분리로 변경·장애·시크릿 경계는 정리됐습니다. 그런데 서버를 나눠도 해결되지 않은 문제가 하나 남아 있습니다. **추론은 여전히 40~50초가 걸린다는 사실입니다.**

### 남은 문제: 분리해도 40~50초는 그대로

사용자가 "인사이트 생성" 버튼을 누르고 응답을 동기로 기다리는 구조라면 두 가지 문제가 생깁니다.

1. **사용자 스레드 점유** — 요청 하나가 서버 스레드를 40~50초씩 붙잡습니다. 스레드풀은 기본 200개라, 동시 요청이 몰리면 스레드가 고갈되어 AI와 무관한 코어 기능까지 전부 응답 불능이 될 수 있습니다.
2. **GPU 서버 과부하** — 로컬 LLM은 동시 추론에 약합니다. 요청이 몰린 만큼 그대로 GPU에 꽂히면 전체가 함께 느려지다 죽습니다.

### 해결: 요청은 큐에 쌓고, 한 건씩 처리한다

그래서 인사이트 생성을 비동기 구조로 바꿨습니다. Kafka를 메시지 큐로 사용해, 요청은 큐에 쌓아두고 코어 서비스가 한 건씩 꺼내 처리합니다.

- 사용자가 요청하면 코어 서비스는 Kafka에 메시지 하나(userId)만 넣고 **즉시 202 "생성 중"을 응답**합니다. 사용자 요청 스레드는 1초 안에 반납됩니다.
- 프론트는 상태 API를 몇 초 간격으로 폴링하다가, DONE이 되면 결과를 조회합니다. 상태는 Redis에 PROCESSING / DONE / FAILED로 기록하고, 같은 사용자의 중복 요청도 이 키로 차단합니다.
- 코어 서비스 안의 Kafka 리스너가 큐에서 메시지를 **한 번에 1건씩** 꺼내, 데이터 조회 → 프롬프트 조립 → ai-app 호출 → 결과 저장까지 처리합니다.

### 이 구조로 얻은 것 3가지

1. **응답성** — 40~50초의 대기가 사용자 요청 스레드에서 사라졌습니다. 아무리 요청이 몰려도 코어에서 추론을 기다리는 스레드는 리스너 1개뿐입니다.
2. **백프레셔** — 동시에 100명이 요청해도 GPU에는 한 번에 한 건씩만 갑니다(토픽을 1파티션으로 고정해 직렬화). 큐가 대신 줄을 세워줍니다.
3. **유실 방지** — 요청이 브로커에 남아 있어서, 처리 중에 서버가 재시작돼도 큐에 쌓인 요청은 사라지지 않습니다.

### 헷갈리기 쉬운 것: 큐에서 꺼내는 건 ai-app이 아니라 코어 자신이다

"AI 작업이니까 ai-app이 큐에서 꺼내가야 하는 것 아닌가?" 처음엔 저도 그렇게 생각했습니다. 그런데 메시지에 든 건 userId 하나뿐입니다. 이 메시지를 받아서 실제로 해야 하는 일을 나열해 보면 답이 나옵니다.

1. 그 사용자의 포트폴리오·배당·설문 데이터 조회 (MySQL)
2. 조회한 데이터로 프롬프트 조립
3. **LLM 추론 ← 여기만 AI의 일**
4. 응답 파싱 → 인사이트로 가공
5. DB 저장

다섯 단계 중 AI가 필요한 건 3번 하나고, 나머지는 전부 코어의 데이터와 코어의 로직입니다. ai-app이 큐를 소비하려면 DB 접근과 도메인 로직을 통째로 가져가야 해서, "추론만 하는 얇은 서버"라는 분리 원칙이 깨집니다. 그래서 큐는 코어가 소비하고, ai-app에는 3번 한 단계(추론)만 HTTP로 위임합니다.

### 한계와 다음 단계

이 설계가 완벽하다는 뜻은 아닙니다. 정직하게 적어두면 다음과 같습니다.

- **처리량 상한** — 한 번에 1건씩 처리하므로 시간당 최대 수십 건 수준입니다. 사용자가 늘면 리스너를 별도 워커로 분리하는 게 다음 단계입니다.
- **프로세스 수준의 격리** — 코어와 ai-app이 같은 호스트의 도커 위에서 돌기 때문에, 하드웨어·자원 수준까지 격리된 것은 아닙니다.
- **검증은 아직** — "AI가 죽어도 코어는 무영향"이라는 주장을 부하테스트로 실측하는 것이 다음 글의 주제입니다.

### 정리

> **분리의 기준은 도메인이 아니라 워크로드의 특성이다.**

같은 "투자 서비스" 도메인 안의 기능이라도, 다음 네 가지가 다르면 다른 집에 사는 게 낫다고 판단했습니다.

- **실행 시간** — ms 단위 API vs 40~50초짜리 추론
- **하드웨어** — 일반 앱 서버 vs GPU 서버
- **비밀값** — 없음 vs API 키·GPU 서버 주소
- **변동성** — 안정된 비즈니스 로직 vs 잦은 모델·프롬프트 교체

그래서 **공간은 ai-app으로 격리했고, 시간은 Kafka로 비동기 처리했습니다.**

다음 글에서는 이 설계가 실제로 약속을 지키는지 — AI가 죽어도 코어는 정말 무사한지 — 부하테스트로 확인해 보겠습니다.
