# MVP 개발 핵심 문제 정리

> kwak-service (inv-back) MVP 개발 과정에서 집중적으로 다룬 핵심 기술 문제들을 정리한 문서입니다.

---

## 1. 실시간 주가 데이터 제공 — 폴링 없이 어떻게 Push할 것인가

### 문제
클라이언트가 주가 변화를 파악하려면 지속적인 API 요청(폴링)이 필요하다. 사용자 수가 늘수록 서버 부하가 기하급수적으로 증가한다.

### 해결 방식
- **SSE(Server-Sent Events)** 기반 단방향 스트림 채택
  - 개별 종목 구독: `/api/v1/stocks/price/stream/{stockCd}`
  - 전체 종목 브로드캐스트: `/api/v1/stocks/price/stream`
- **30초 하트비트**로 유휴 연결 유지 및 끊긴 연결 감지
- SSE timeout을 `-1`로 설정해 `AsyncRequestTimeoutException` 방지

### 교훈
폴링 대신 Push 방식으로 전환하면 클라이언트-서버 간 불필요한 왕복을 제거할 수 있다.

## 2. 캐시 계층 설계 — Redis vs Caffeine, 무엇을 어디에 쓸 것인가

### 문제
주가처럼 초당 수십~수백 번 읽히는 데이터를 매번 DB에서 조회하면 성능 병목이 발생한다. 그렇다고 모든 캐시를 Redis에 넣으면 네트워크 레이턴시가 생긴다.

### 해결 방식

| 캐시 | 적용 대상 | 이유 |
|------|-----------|------|
| **Caffeine** (인메모리) | 실시간 주가 (최대 5,000 종목, TTL 6시간) | 나노초 수준 조회, 네트워크 오버헤드 없음 |
| **Redis** | JWT 세션, 토큰 블랙리스트 | 서버 재시작 후에도 유지, 다중 서버 공유 가능 |

### 주가 조회 폴백 흐름
```
Caffeine 캐시 → (미스 시) DB 조회 → (없으면) null 반환
```

### 교훈
데이터 특성(갱신 빈도, 공유 범위, 내구성)에 따라 캐시 계층을 분리해야 한다.

## 3. JWT 보안 설계 — 토큰 탈취 시 피해를 최소화하는 방법

### 문제
JWT에 사용자 정보를 담으면 편리하지만, 토큰이 탈취되면 사용자 신원이 즉시 노출된다.

### 해결 방식
- JWT payload에는 **sessionId(UUID)만** 저장
- 사용자 정보(userId, nickname, email, 프로필 이미지)는 **Redis에만** 저장
- 토큰 탈취 시 공격자는 세션 ID만 알 수 있고, Redis 없이는 사용자 특정 불가

```
JWT (sessionId only) → Redis 조회 → 실제 사용자 정보 반환
```

### 추가 고려
- `Authorization: Bearer` 헤더 + `accessToken` 쿠키 **이중 인증 경로** 지원
  - SPA(헤더 방식)와 전통적 웹(쿠키 방식) 모두 수용
- 로그아웃 시 **토큰 블랙리스트** Redis에 등록

### 교훈
JWT payload 최소화는 단순한 베스트 프랙티스가 아니라, 실제 공격 시나리오를 고려한 설계 결정이다.

## 4. 멀티 서비스 인증 — 서비스 간 호출 없이 JWT를 독립적으로 검증하는 방법

### 문제
마이크로서비스 환경에서 각 서비스가 인증을 위해 중앙 서버에 매번 요청하면 단일 장애점(SPOF)이 생기고 레이턴시가 증가한다.

### 해결 방식
- **공통 라이브러리(`common` 모듈)** 로 JWT 검증 로직을 공유
- 각 서비스가 공통 모듈의 `JwtProvider`를 사용해 **독립적으로 토큰 검증**
- 서비스 포트 구성:

| 서비스 | 포트 | 역할 |
|--------|------|------|
| portal | 8080 | 메인 (인증, 포트폴리오, 주가) |
| survey-service | 8081 | 설문 관리 |
| stock-advisor | 8082 | 투자 성향 분석 |
| market-analyzer | 8083 | 시장 통계 |

### 교훈
공통 라이브러리를 통한 인증 공유는 서비스 간 결합도를 낮추면서도 일관성을 유지하는 방법이다.

## 5. 포트폴리오-거래내역 동기화 — 데이터 정합성 유지 문제

### 문제
거래(매수/매도) 발생 시 포트폴리오 아이템이 자동으로 갱신되지 않으면 보유 수량과 실제 거래 이력 간 불일치가 생긴다.

### 해결 방식
- 거래내역 생성 시 포트폴리오 아이템 **자동 동기화** 처리
- 매수 → 보유 수량 증가, 매도 → 수량 감소 (0이면 아이템 삭제)

### 교훈
관련 도메인 간 상태 일관성은 애플리케이션 레이어에서 명시적으로 관리해야 한다.

## 6. 투자 성향 분석 — 설문 응답 기반 리스크 프로파일링

### 문제
단순 설문 결과를 저장하는 것을 넘어, 사용자의 응답을 바탕으로 의미 있는 투자 성향(위험 선호도, 관심 섹터)을 도출해야 한다.

### 해결 방식
- **리스크 프로파일**: 설문 응답을 집계해 보수적/중립/공격적 투자 성향 분류
- **선호 섹터 분석**: 전체 포트폴리오의 매수 금액을 섹터별로 집계
- **시장 리스크 비교**: 개인 투자자 vs 전체 사용자 평균 비교 제공

### 기술적 주의점
- `GROUP BY` 없이 집계 함수 사용 시 데이터 정합성 오류 → 쿼리 수정으로 해결

---

## 7. 환경 분리 — 로컬 개발과 프로덕션 설정의 완전한 분리

### 문제
개발 편의를 위한 설정(Swagger, 하드코딩된 자격증명)이 프로덕션에 노출되면 보안 위협이 된다.

### 해결 방식
- Spring Profile 기반 분리: `local` / `prod`
- **로컬**: 하드코딩 자격증명, Swagger UI 활성화, localhost URL
- **프로덕션**: 환경변수 주입 (`DB_HOST`, `REDIS_HOST`, `JWT_SECRET` 등), Swagger 비활성화
- Docker 기반 서비스별 독립 배포 지원
